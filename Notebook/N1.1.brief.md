## 🏗️ Architecture Ki Briefing (Humne Kya-Kya Banaya Hai)

Hamara poora system 3 bade bhaagon mein bata hua hai jo aapas mein real-time baat karte hain:

### 1. Engine Loop (`miller_core`)

* **Kaam:** Yeh aapke pure system ka main dimaag hai.
* **Status:** Yeh user se prompt leta hai, local Qdrant database se purana relevant code context nikaalta hai, aur `Qwen-3B` model se naya code generate karwata hai.
* **Self-Healing Layer:** Agar generated code mein koi syntax galti hoti hai, toh yeh `rustc` ka error khud padhta hai aur bina aapko pareshan kiye **5 attempts** tak use khud thandey dimaag se repair karne ki koshish karta hai.
* **Sandbox Safety:** Yeh code ko direct chalane ke bajaye ek alag file `sandbox.rs` mein run karta hai taaki aapka real codebase safe rahe.

### 2. AST Graph Parser (`miller_parser`)

* **Kaam:** Yeh aapke code ki "Aankhein" hai jo code ko sirf text ki tarah nahi padhti, balki uske structure ko samajhti hai.
* **Status:** `tree-sitter` library ka use karke yeh kisi bhi `.rs` file ke andar ghuskar dhoodhta hai ki usme kaun sa **Struct** hai, kaun sa **Function** hai, ya kaun sa **Enum** hai. Yeh unhe alag-alag tukdon (chunks) mein tod deta hai taaki database mein saaf data jaye.

### 3. Smart Vector Memory Layer (`miller_memory`)

* **Kaam:** Yeh Miller ki "Yaaddasht" (Memory) hai.
* **Status:** Local **Qdrant Vector Database** aur Ollama ke `all-minilm` embedding model ka use karke, yeh parser dwara tode gaye code chunks ko mathematical vectors mein badal kar permanent store karta hai. Jab aap bolte ho "purana car wala code badlo", toh yeh yahin se dhoodh kar lata hai.

## 🔄 Pichli Do-Teen Chats Mein Hum Kahan Phase Aur Kya Badla?

Humne jab single-file se upar uth kar **Multi-File Project Scanning** par kaam shuru kiya, toh pichle 2-3 turns mein humne do bade challenges ko face kiya aur unhe solve karne ki koshish ki:

1. **The '0 Entities' Mismatch:** Shuruat mein jab humne poora folder scan kiya, toh parser ne `0` entities dikhayi. Uska reason yeh tha ki hamari Tree-sitter query sirf top-level functions ko pakad rahi thi, jabki hamara asli code `impl` blocks (methods) ke andar tha. Humne query ko upgrade kiya taaki wo methods ko bhi pakde.
2. **The "Cursor Mode" Background Idea:** Pehle humne ek `scan` command banayi thi, par aapne bilkul sahi point pakda ki Cursor mein user ko alag se scan nahi likhna padta. Isliye humne scanner ko **Tokio Background Thread (`tokio::spawn`)** par dala taaki Miller start hote hi chupchaap background mein apna kaam kare aur aapko instantly prompt mil jaye.
3. **Incremental Hashing (RAM Protection):** 8GB RAM par 1000+ files ko baar-baar scan karne se Ollama crash na ho, isliye humne **SHA-256 Hashing** ka logic socha, taaki jo file badli nahi hai use instantly skip kiya ja sake aur system par load $0\%$ ho jaye.

## 📊 Miller vs Cursor: Kahan Same Hain Aur Kahan Alag?

Hum dono ke system-level approach ko inhi 4 pillars par compare karte hain:

### 1. Persistent File Watcher

* **Abhi Miller kahan same hai:** Dono ka goal same hai—pure project ko baar-baar scan na karna pade.
* **Farq (Difference):** * **Cursor** OS-level background event listener (`inotify` on Linux) ka use karta hai. Jab tak Cursor khula hai, OS use live signal bhejta rehta hai ki *"Bhai, `main.rs` save hui hai, dekh le"*.
* **Miller** abhi ek **Polling-based background thread** use kar raha hai jo program shuru hote hi chalti hai. Yeh live OS signals ko nahi sunta, balki startup par folder check karta hai.


* **Kya karna baki hai:** Future mein humein Miller mein `notify` crate add karni padegi, taaki Miller ek persistent daemon (background service) ban sake jo file save hote hi automatic trigger ho jaye.

### 2. Incremental Indexing & Hashing

* **Abhi Miller kahan same hai:** **Yahan hum bilkul Cursor ke raste par hain!** Jo logic humne pichle turn mein design kiya (SHA-256 Hashing aur `.miller_cache.json`), wo bilkul wahi kaam karta hai jo Cursor karta hai.
* **Farq (Difference):**
* **Cursor** local embedded DB (**LanceDB**) use karta hai jo seedhe vector memory aur hashes ko file system par fast track karti hai.
* **Miller** hashes ko ek simple `.json` state file mein rakhta hai aur vectors ko local **Qdrant (Docker)** container mein bhejta hai. Data structures ke mamle mein hamara approach bhi bohot clean hai.


* **Kya karna baki hai:** Hamara hashing logic paper par ready hai, use code file mein perfectly chala kar verify karna baki hai taaki `Skipped (No Changes)` wala counter hamesha zero (0) na aaye.

### 3. LSP (Language Server Protocol) Integration

* **Abhi Miller kahan same hai:** **Yahan bada gap hai.** Miller abhi isme Cursor se alag hai.
* **Farq (Difference):**
* **Cursor** compiler ke dimaag (`rust-analyzer`) se direct baat karta hai. Agar code mein error hai ya koi cross-reference dhoodhna hai, toh wo LSP se types aur definition paths uthata hai.
* **Miller** poori tarah se **Tree-sitter (AST Parsing)** aur **Semantic Vector Search (Qdrant)** par nirbhar hai. Miller code ke text aur syntax structure ko samajhta hai, par compiler ke real-time contextual relations ko direct query nahi kar pata.


* **Kya karna baki hai:** Yeh hamare roadmap ka ek bada feature hoga. Humein `jsonrpc` ka use karke local `rust-analyzer` se Miller ko connect karna hoga taaki wo types ki definition accuracy ko 100% sahi pakde.

### 4. Myers Diff Engine (Local Apply)

* **Abhi Miller kahan same hai:** **Yahan dono poori tarah alag hain.** * **Farq (Difference):**
* **Cursor** sirf badli hui lines ko inject karta hai (`git diff` style mein). Agar 1000 lines ki file mein 2 lines badalni hain, toh wo sirf un do lines ko badalta hai.
* **Miller** abhi poori tarah **Isolated File Sandbox** use karta hai. Wo `sandbox.rs` naam ki nayi file banata hai aur usme poora code naye sire se likhta hai. Wo purani file ke andar line-by-line diff apply nahi karta.


* **Kya karna baki hai:** Humein Miller mein ek local diff library (jaise `similar` crate) lagani hogi, taaki jab Miller kisi complex multi-file project ko edit kare, toh wo poori file delete karke na likhe, balki sirf specific functions ko target karke patch/diff apply kare.
